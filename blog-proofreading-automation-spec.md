# ブログ校閲・公開自動化 たたき台

## 目的

- ブログ新規投稿後に自動で校閲する
- 校閲後に自動修正する
- 公開可能な品質まで整えたうえで公開する

## 校閲の優先事項

優先度が高い項目:

1. 誤字脱字
2. 防止したい表現の検出と修正
3. 読みやすさの改善

公開基準:

- 校閲後、そのまま公開できる状態を目指す

## チェック対象

- 誤字脱字
- 不自然な日本語
- 同じ言い回しの繰り返し
- 事実として危ない表現
- 他社記事に似すぎていないか

修正方針:

- かなり自然な文章に整える
- 文体は親しみやすくする
- 「です・ます調」で統一する
- 文の長さは普通

## NG表現ルール

- 過度に煽る言い方は避ける
- 断定しすぎる表現は避ける
- 価格に関する表現は避ける

補足:

- 外壁塗装系の記事も想定
- 個別の禁止語は今後追加予定

## SEO・構成ルール

- SEOキーワードは残したまま自然に整える
- 地域名は残す
- 見出しにもキーワードを入れる
- 構成も見直す
- 問い合わせ導線は毎回必須
- 見出し形式の固定ルールは現時点ではなし

## 類似対策

- 他社記事との類似チェックを行う
- 自社過去記事との重複チェックを行う
- 類似していた場合は書き直し候補を出す
- 類似箇所だけ修正する

## 出力形式

- 問題点一覧を出す
- 指摘は厳しめ
- 点数評価は不要

## 自動化フロー

想定フロー:

1. 新規ブログ記事を受け取る
2. 校閲ルールに沿ってチェックする
3. 問題点一覧を生成する
4. 自動で本文を修正する
5. 修正後本文を別セルまたは別フィールドに保存する
6. 公開条件を満たした場合にブログ公開する

実行タイミング:

- 記事生成の直後

エラー時の扱い:

- 原則は自動修正

テスト方針:

- まずは1記事でテスト

## 接続先の確定内容

- 記事生成元:
  - WordPress の下書き投稿
  - WordPress の予約投稿
- 公開先:
  - WordPress
- 類似チェック対象:
  - すべて
  - 自社過去記事
  - 外部公開済み記事

## 実装前に残る確認事項

- 校閲に使うAIの接続先
  - 例: OpenAI API
- 外部記事の類似チェック方法
  - 例: 登録済み競合URL群との比較
  - 例: 検索API連携で上位記事を取得して比較
- 公開前に人の確認を完全に省くか
  - 省く場合は公開判定ルールを厳しめにする

## 仮の公開判定ルール案

- 誤字脱字の未解決件数が0
- NG表現の未解決件数が0
- 危険表現が警告以下
- 類似率がしきい値未満
- 問い合わせ導線が本文内に存在する

## 実装イメージ

候補:

1. WordPress下書き / 予約投稿起点 + AI校閲 + 自動更新 + 公開
2. WordPress下書き / 予約投稿起点 + AI校閲 + 類似チェック + 公開判定 + 公開

今回の要件では、WordPress を起点にした構成が最も自然。

## 推奨フロー

1. WordPress に新規記事が下書きまたは予約投稿で作成される
2. 自動処理が対象記事を検知する
3. 本文とタイトル、見出し、メタ情報を取得する
4. 校閲AIで問題点一覧を作成する
5. 校閲AIで本文を自動修正する
6. 類似チェックを行う
7. 類似箇所があれば該当箇所だけ再修正する
8. 問い合わせ導線の有無、NG表現、危険表現を最終確認する
9. 修正版を WordPress の別カスタムフィールドまたは別本文領域に保存する
10. 公開条件を満たした場合のみ公開する
11. 条件を満たさない場合は下書きに戻し、問題点一覧を保存する

## WordPress 前提の実装案

### 案A: WordPress プラグイン方式

特徴:

- WordPress 内で完結しやすい
- 下書き保存や予約投稿時にフックを使える
- 投稿メタに校閲結果を保存しやすい

使うもの:

- `save_post`
- `transition_post_status`
- `wp_update_post`
- 投稿メタ保存
- WP-Cron または外部 Cron

向いているケース:

- WordPress 側だけで運用したい
- サーバーに PHP 実装を置ける

### 案B: 外部ワーカー方式

特徴:

- AI呼び出しや類似チェックを拡張しやすい
- 複雑な判定を管理しやすい
- WordPress は REST API で連携する

使うもの:

- WordPress REST API
- 外部スクリプト
- 定期実行
- ログ保存

向いているケース:

- 類似チェックを強めにやりたい
- 将来複数サイトへ広げたい

## 推奨方式

最初の1記事テストでは、以下を推奨:

- WordPress 下書き / 予約投稿を取得
- AI校閲で本文を修正
- 自社過去記事との類似チェックを実施
- 問題がなければ公開
- 外部記事との類似チェックは第2段階で追加

理由:

- まずは WordPress と校閲ロジックの安定化を優先できる
- 外部記事の全件比較は実装コストと運用コストが高い
- 「すべて」を初回から完全自動にすると誤判定リスクが高い

## 公開判定ルールの推奨値

- 誤字脱字の指摘が未解決でない
- NG表現の指摘が未解決でない
- 断定的な危険表現が残っていない
- 問い合わせ導線が本文末に存在する
- 同一見出しや同一表現の連続が許容値以内
- 自社過去記事との類似がしきい値未満
- 外部記事との類似が警戒しきい値未満

## 保存先の推奨

- 元本文:
  - WordPress 本文
- 校閲後本文:
  - カスタムフィールド
- 問題点一覧:
  - カスタムフィールド
- 公開判定:
  - カスタムフィールド
